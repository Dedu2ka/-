# FSA

##  Порядок работы с приложением (Инструкция пользователя)

### Этап 1. Построение деревьев функций и модулей ПО
При старте приложения на главном экране пользователь параллельно выстраивает две независимые иерархические структуры проекта:
1. **Функциональная модель (Дерево функций):** Сюда вносятся пользовательские требования (главные, основные и вспомогательные функции). Каждой функции экспертным путем присваивается вес (локальная значимость). Программа рассчитывает глобальную значимость, приводя сумму долей конечных листьев строго к $100\%$.
2. **Структурная модель (Дерево модулей):** Сюда вносится программная архитектура (компоненты, классы, подсистемы, СУБД). Каждому модулю нижнего уровня присваивается стоимостной вес его разработки (`CostWeight`), сумма которых на уровне листьев также нормируется к $100\%$.

### Этап 2. Заполнение матрицы связей
В окне анализа (`AnalysisDiagramWindow`) пользователь заполняет функционально-структурную матрицу связей (N-Square Matrix). На пересечении строк и столбцов указывается, какие программные модули задействованы в обеспечении конкретных функций. 
* На основе связей алгоритм производит автоматический математический перенос стоимостей с компонент структуры на функции и перенос значимостей с функций на компоненты.

### Этап 3. Анализ совмещенной диаграммы значимости и стоимости
После нажатия на кнопку расчетов открывается окно совмещенного анализа (`FunctionSignificanceDiagramWindow`), в котором осуществляются:
1. **Визуальный контроль гистограмм:** Для каждого листа нижнего уровня строятся два параллельных столбца: столбец значимости/полезности (Синий или Оранжевый) и столбец затрат (Ярко-красный).
2. **Интерактивная навигация:** Графики ScottPlot поддерживают свободное перемещение (зажатием ЛКМ/ПКМ) и масштабирование (колесиком мыши). Чтобы сбросить зум, нажмите ПКМ на графике и выберите *Auto Axis*.
3. **Получение экспертного заключения:** По нажатию на кнопку **«Сгенерировать советы ФСА»** запускается встроенная экспертная система, оценивающая качество спроектированной программной архитектуры.

## Интерпретация автоматических советов и индексов ФСА

Над каждой парой столбцов вычисляется **Индекс ФСА** ($I$) по формуле:
$$I = \frac{\text{Significance (Значимость)}}{\text{CostWeight (Доля затрат)}}$$

В выпадающем экспертном отчете программа группирует дефекты по жестким методологическим критериям:

| Критерий в коде | Выдаваемый вердикт программы | Что это означает по теории ФСА / Совет архитектору |
| **Сумма весов $\neq 100\%$** | `[ОШИБКА НОРМИРОВАНИЯ]` | Нарушен фундаментальный закон сохранения 100% долей. Требуется пересчитать локальные матрицы сравнения. |
| **Индекс $I < 0.85$** | `[ПЕРЕРАСХОД]` | Доля затрат на написание кода существенно превышает реальную ценность функции для пользователя. **Совет:** упростить алгоритм, вырезать избыточную реализацию. |
| **Индекс $I > 1.25$** | `[ЗОНА РИСКА]` | Сверхзначимая для пользователя функция стоит «подозрительно дешево». **Совет:** высок риск скрытого технологического долга или багов. Проверьте надежность этого модуля. |
| **Затраты $= 0\%$** | `[КРИТИЧНО]` | Функция заявлена в требованиях, но полностью забыта программистами и не привязана ни к одному модулю кода. |
| **Модуль имеет `functions = 0`**| `[ДЕФЕКТ СВЯЗЕЙ]` | Программный модуль стоит денег, но не выполняет ни одной полезной функции. **Совет:** применить правило псевдомодуля $M_0$ (выделение системного каркаса) или распределить затраты. |
| **Перерасход в $>30\%$ элементов** | `Эффект второй системы Брукса` | Интегральный дефект. Архитектор увлекся сложной и дорогой разработкой малозначимых «фич». Требуется немедленный рефакторинг спецификаций. |
